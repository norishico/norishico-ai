# 12人委員会 議論手続き

## メンバー（役割で発言させる。口調詳細はメモリ project_norishiko.md「委員会 12人体制」）
みなみ(データ) / れいな(現場) / ゆきこ(リスク) / さくら(血統) / あかり(オッズ) / ひなた(展開) / あおい(騎手) / まなつ(ML) / りこ(UI) / かえで(ベイズ) / りさ(エンジニア) / ゆめ(元厩舎)

## 禁止事項
- 「全員一致で◎」「委員会合意」で直進すること（禁止ワード）
- 12人の口調で自分の案を美化・正当化すること（全員に賛成させるのは演出であって議論ではない）
- 反対意見ゼロでの実装進行

## 実装前の強制チェックリスト（1つでも欠けたら実装しない）
1. **ユーザーシナリオ3本** — 初見ユーザーがこの通知/画面/変更を見て何を感じるか、具体的に1行ずつ
2. **反対意見 最低1件** — 12人から1人を指名し、本気の反対立場で書く。反対が出ない案は設計が浅い
3. **不利益シナリオ1件** — この実装で困るユーザーの具体例（例:「発走5分前にスマホを開いた人は通知過多で本命を見逃す」）
4. **のりお実装前確認** — 上記3点を揃えて方針説明し、承認を得てから実装

## 実装後の流れ
1. /bt-verify で4年Walk-Forward CV（通常+Winsorized 5万円キャップ）検証
2. 結果を12人に評価させて最終判断（ここでも反対意見必須）

## ゆきこの過学習防御5原則（スコアリング変更時は必ず通す）
1. 時系列CV必須 2. 2026をholdout 3. SHAP値確認 4. Kelly基準 5. v6.6常時ベースライン比較

## 設計時の観点（過去の叱責4件からの教訓）
- 通知は「誰が/いつ/何を見るか」で設計（開発者視点でなくユーザー視点）
- 前日情報は「今日の行動に使えるか」で判断（情報は量より配達タイミング）
- 予想公開と最終確定を混同しない
- 複数日データを持つJSONから通知するなら日付フィルタ必須
