# 🛡 v6.6 実運用 障害リスト & チェックリスト

作成: 2026-04-14
作者: れいな (現場観察) + 9人委員会レビュー
対象: のりお (ユーザー) + システム運用
目的: バックテスト ROI 129.3% が**実運用で再現されない原因を先回りで特定**し、対策を準備する

## 🚨 想定される障害パターン

### カテゴリ A: オッズ関連障害（最重要）

#### A1. 発走直前のオッズ変動で買い条件から外れる
**症状**: 朝の予想では買い対象だったが、発走時に範囲外
**発生場面**:
- 3勝クラス 8-11倍 → 発走時 7.8倍 or 11.2倍
- F1未勝利 15-33倍 → 発走時 14.5倍 or 33.5倍
- G1/G2 7-10倍 → 発走時 10.3倍

**対策**:
- ✅ auto_refresh の ±20%ロック機構で大半カバー
- ⚠️ **それでも乖離20%超えの馬は買えず** → 発生率計測する
- 📊 **ログ項目**: 朝判定時オッズ / 発走時オッズ / ロック発動有無

#### A2. 人気馬の過熱（オッズ急落）
**症状**: 朝8倍→発走時4倍 (50%下落、ロック解除)
**発生場面**: 
- 重賞本命の人気化
- 競馬ファンの直前情報共有

**対策**:
- ロック解除されたら原則スキップ (±20%超えは統計的に買わない設計)
- 📊 **ログ項目**: ロック解除発生数

#### A3. オッズ取得失敗
**症状**: Selenium エラー、netkeiba のレート制限
**発生場面**: 
- 過去の事例: Chromeタブクラッシュ
- 大量アクセス時の 403/429 エラー

**対策**:
- ✅ auto_refresh はエラー時に次のトリガーへ継続
- ⚠️ **連続失敗時のアラート**が未実装 → 追加必要
- 📊 **ログ項目**: fetch失敗回数/時刻

---

### カテゴリ B: 購入実行障害

#### B1. 馬券購入のタイミング遅れ
**症状**: のりおが予想を見て馬券を買う前に発走
**発生場面**:
- 10:05 発走の 1R → 朝9時半に予想見ても9:55に買わないといけない
- 複数会場同時発走で対応遅れ

**対策**:
- UI に「次の買い候補レース」カウントダウン表示？
- 通知 (note連携で毎朝メール？)
- **注意**: これはのりおの運用習慣に依存

#### B2. 馬券種類のミス
**症状**: 単勝買うつもりが馬連、金額間違い
**発生場面**:
- 8倍以上: 単勝500+馬連1500
- 8倍未満: 単勝1000+馬連1000
- challenge: 単勝1000のみ
- special: 単勝1000のみ

**対策**:
- ✅ HTML 予想カードに「買い目」明記済み
- 📊 **ログ項目**: のりおの実際の買い目を記録 (手動)

#### B3. 資金不足
**症状**: 週末に複数レース該当 → 想定投資超過
**発生場面**:
- 1日 5-6R 買いになった日の総額が想定超え
- v6.6 平均 週末 4-5R × 2000円 = 8-10,000円

**対策**:
- ✅ HTML サマリーに「想定投資」表示済み
- 資金管理はユーザー責任

---

### カテゴリ C: データ品質障害

#### C1. 結果取得の遅延・失敗
**症状**: 発走後の結果が monthly_results に反映されない
**発生場面**:
- auto_refresh の save-results 実行失敗
- netkeiba の結果確定遅延

**対策**:
- ✅ 手動で `publish_weekend.py --save-results YYYYMMDD` で再実行可能
- ⚠️ **自動リトライ機構が未実装**

#### C2. 結果の誤記録
**症状**: save_results.py の日付バグ等で別日に記録
**発生場面**:
- 過去に発生: datetime.now() で深夜跨ぎ時の誤認
- ✅ 修正済み (引数 or ファイル名から日付抽出)

#### C3. スコアリングキャッシュ汚染
**症状**: キャッシュが前週データで汚染されて新週の予想精度低下
**発生場面**:
- prefetch_month が月を跨いだ時の挙動
- 過去のaccel_lapバグ等

**対策**:
- ✅ 決定論性テスト (backtest_v6.py で再現性確認)
- ⚠️ **本番側の定期的な正気度テスト未実装**

### カテゴリ D: システム稼働障害

#### D1. auto_refresh が止まる
**症状**: プロセスクラッシュ、スリープモード等
**発生場面**:
- Windows のスリープ / シャットダウン
- メモリ不足
- 過去の事例: Chromeタブクラッシュ

**対策**:
- ✅ Python プロセスの生存監視 (手動)
- ⚠️ **自動再起動機構がない**
- 📊 **ログ項目**: auto_refresh 稼働時間、中断回数

#### D2. git push 失敗で本番反映されない
**症状**: ローカル更新は反映されるが docs/ が GitHub Pages に push されない
**発生場面**:
- ネットワーク断
- git 認証切れ
- 過去の事例: 19コミット push 忘れ

**対策**:
- ⚠️ **push 確認**が毎回手動
- 📊 **ログ項目**: 最終push時刻、未push commits数

#### D3. GitHub Pages のキャッシュ
**症状**: docs 更新後も旧ページが表示
**発生場面**: GitHub Pages の遅延

**対策**:
- ブラウザのハードリロード (Ctrl+F5) で解決
- バージョン番号 or タイムスタンプをフッターに

---

### カテゴリ E: 判定ロジック障害

#### E1. 極端な外れ値レース
**症状**: バグではないが、単体で大損失 (-10,000円超え等)
**発生場面**:
- 大穴的中(+60,000) を外した瞬間
- 本命が多頭数で不利枠

**対策**:
- ✅ 不利枠フィルタ (新潟芝後半内、札幌芝後半内、中山ダ前半内)
- モニタリングで週次確認

#### E2. 想定外のレース種別
**症状**: 障害レース、交流重賞等を誤認
**発生場面**:
- ✅ 障害レースは `'障害' in race_name` で除外済み
- ⚠️ 地方招待レース、国際競走の特殊扱い

**対策**:
- predict_weekend.py で除外ルール確認

#### E3. ev_okだが買えないケース
**症状**: is_buy_v6 でTrueだが、何らかの理由で HTML に出ない
**発生場面**: 過去のstale_mode判定等

**対策**:
- ✅ stale_mode検知追加済み
- 📊 **ログ項目**: ev_ok件数 vs HTML表示件数の一致確認
