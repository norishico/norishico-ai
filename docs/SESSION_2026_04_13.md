## 🚨 発見・修正した重大バグ

### バグ#1: accel_lap 常時False（最重要）
- **場所**: backtest_2026.py prefetch_month
- **症状**: `_training_actual_cache` に `accel_lap` キーが格納されておらず、score_training_actual のキャッシュヒット時に常にFalseを返す
- **影響**: C2新馬・F1未勝利ルールが**全年完全に0件発動**
- **修正**: prefetch_month の SELECT に `lap2` を追加、キャッシュ格納時に `accel_lap = lap1 < lap2` を設定
- **効果**: ROI +13.2pt (75.4→88.6%)

### バグ#2: race_name キャッシュ漏れ
- **場所**: backtest_2026.py prefetch_month (同じ関数)
- **症状**: prefetch SELECT に race_name がなく、scoring.py L517 の `_grade_from_race_name(row.get('race_name', ''))` が常に空文字列を返す
- **影響**: 過去走スコアの上がり3F成分のクラス別平均が誤ったクラスで計算され、honmei選定精度が低下
- **修正**: prefetch SELECT に race_name を追加
- **効果**: 2024年単独で +28,380円改善（v6本体ROI 90.7→96.3%）

### バグ#3: score 非決定論性
- **場所**: backtest_2026.py score_one_race の res.sort() 3箇所
- **症状**: 同点ブレイク時に Python の安定ソートが入力順依存 → 同一条件の複数実行で毎回結果が微妙に違う (件数1-2R、損益数千円の揺れ)
- **原因**: total_score が同じ馬の並び順が、キャッシュ構築の副作用で実行間で変動
- **修正**: ソートキーに馬名を追加: `sort(key=lambda x: (-x['total_score'], x['horse_name']))`
- **効果**: 2024年2回連続実行で完全一致を確認

### 追加: predict_weekend.py も同じprefetch_monthを使うため、バグ#2と#3の修正は本番推論にも波及

## 🎓 得られた教訓

1. **キャッシュ格納フィールドは scoring側が期待する全フィールドを網羅せよ**
   - accel_lap と race_name の2バグは同じ構造。prefetch SELECT と scoring fallback SELECT の整合性チェックを仕組み化すべき

2. **変更は1つずつ検証する**
   - Phase 2 でC2拡張+F1拡張+G2絞込を同時実行して失敗。個別検証したらC2拡張のみ-100,400で足を引っ張っていたと判明

3. **実運用現実性を常に考慮する**
   - のりおの「1勝4.0-4.5倍は買えない」指摘でバンド幅問題を発見。バックテストの数値に惑わされず、オッズ変動幅との相対関係を見るべき

4. **決定論性は品質の基礎**
   - 同じ条件で同じ結果にならないと検証作業自体が信頼できない。ソートキーに安定キーを入れる習慣を

5. **重要ファイルはgit追跡必須**
   - backtest_2026.py がgit追跡外だったため過去バージョン復元不可能。CLAUDE.md の数値と現行コードの乖離原因調査が困難だった

6. **CV は頑健性チェックの強力な武器**
   - Walk-Forward CV (学習期間→テスト年) で過学習を防止。2勝廃止・3勝8-11絞込は6/6完全正解で頑健性確認
